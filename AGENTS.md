## Build, Run, Test, Lint

### Setup and development

```bash
composer setup        # install deps, create .env, key:generate, migrate, npm install, npm build
composer dev          # php artisan serve + queue + pail + vite
composer dev:ssr      # SSR mode (includes inertia:start-ssr)
npm run dev           # Vite only
```

### Build

```bash
npm run build         # Production assets
npm run build:ssr     # Production assets + SSR bundle
```

### Tests

```bash
composer test                         # config:clear + php artisan test
php artisan test                      # all tests
./vendor/bin/phpunit                  # all tests via phpunit

./vendor/bin/phpunit tests/Unit/ExampleTest.php             # single file
./vendor/bin/phpunit --filter test_that_true_is_true        # single test method
php artisan test --filter ExampleTest                        # single class / name filter
./vendor/bin/phpunit --testsuite Unit                        # unit suite
./vendor/bin/phpunit --testsuite Feature                     # feature suite
```

### Lint, format, static checks

```bash
vendor/bin/pint             # format PHP
vendor/bin/pint --test      # check PHP formatting only
npm run lint                # ESLint (auto-fix enabled)
npm run format              # Prettier write (resources/)
npm run format:check        # Prettier check
npm run types               # TypeScript no-emit type check
```

### Docker / Makefile shortcuts

```bash
make dev                    # docker compose up --build
make build                  # docker compose build --no-cache
make up && make down        # start / stop containers
make shell                  # shell in app container
make artisan cmd='migrate'  # run artisan in container
```

## CI Parity

- `tests.yml`: runs `./vendor/bin/phpunit` on push/PR to `develop` and `main`
- `lint.yml`: runs `vendor/bin/pint`, `npm run format`, `npm run lint`
- Before finalizing changes, run: `vendor/bin/pint`, `npm run lint`, `npm run format`, `npm run types`

## Code Style and Conventions

### General formatting

- `.editorconfig`: UTF-8, LF, 4 spaces, final newline, trim trailing whitespace
- YAML uses 2 spaces; Markdown keeps trailing whitespace
- Keep files ASCII unless the file already requires Unicode content

### PHP (Laravel)

- Formatter: Laravel Pint default preset (no custom `pint.json`)
- Keep controllers thin; put business logic in models/services when complexity grows
- Use explicit parameter and return types on controller methods
- Validation is done inline with `$request->validate([...])`
- For validation/business failures, return `back()->withErrors(...)`
- For successful forms, use flash data via `back()->with('success', ...)`
- Models use `$fillable` (not `$guarded`) and `$casts` arrays
- Relationship methods are type-hinted (`HasMany`, etc.)
- Route model binding uses slug where defined (`getRouteKeyName(): 'slug'`)
- Imports: one `use` per line, no unused imports, keep ordering consistent (framework then app)
- Naming: classes `StudlyCase`, methods/properties `camelCase`, DB columns `snake_case`

### TypeScript / React / Inertia

- Prettier: single quotes, semicolons, print width 150, tab width 4
- Prettier plugins auto-organize imports and Tailwind classes
- ESLint: `@eslint/js` + `typescript-eslint` + `react` + `react-hooks`
- TypeScript is strict (`strict`, `noImplicitAny`); avoid `any`
- Use `@/*` alias for `resources/js/*`
- Prefer function components with typed props (avoid `React.FC`)
- Define interfaces/types near top of file
- Use Inertia `useForm` for forms and server-side validation errors
- Access translations via `usePage().props.translations` (commonly aliased as `t`)
- Never hardcode user-facing copy when a translation key should be used
- Respect RTL: check locale and apply `dir="rtl"`/layout adjustments

## Architecture and Data Flow

- Flow is server-driven via Inertia: route -> controller -> Inertia page props
- No standalone API layer unless explicitly introduced
- Keep HTTP concerns in controllers; avoid mixing view concerns into models
- If adding a service/repository layer, keep it minimal and consistent with existing code

## Generated or Sensitive Areas

- Do not manually edit generated frontend artifacts:
    - `resources/js/actions/*`
    - `resources/js/routes/*`
    - `resources/js/wayfinder/*`
    - `resources/js/components/ui/*`
- Avoid committing secrets (`.env`, credentials, API keys)

## Agent Workflow Expectations

- Follow existing local patterns first; do not do broad refactors unless requested
- Make focused changes and update/add tests when behavior changes
- Prefer smallest safe diff; avoid touching unrelated files
- Run relevant checks for changed areas before finishing
